# Tracker: Notion

The Notion backend for the engineering skills. Issues and PRDs for this repo live in Notion databases, reached through the Notion MCP. The content skills (`/to-prd`, `/to-issues`, `/triage`) own the *what* — templates, the state/category vocabulary, decision logic, agent briefs. This skill owns the *how*: it translates their canonical artifacts into concrete Notion MCP calls, and is the only place that knows the database IDs, property mappings, and MCP verbs.

It does **not** explore codebases, write templates, or decide triage state — those belong to the content skills. It takes an artifact (or a read / transition request) and talks to Notion.

## Where the IDs live

All database IDs and property mappings live in **`docs/agents/workflow-config.md`** (gitignored — it holds the private workspace IDs). Read that file once at the start of any operation and cache the IDs + property mappings for the conversation. If it is missing, stop and ask the user to recreate it before proceeding.

**Never hardcode a Notion ID — always resolve it from `workflow-config.md`.**

## The AI disclaimer

Every triage comment or brief body this skill writes **must** start with:

```
> *This was generated by AI during triage.*
```

This skill owns prepending it: the content skill hands over the note body, and this skill adds the line.

## Operations

### publishPRD(artifact)

`/to-prd` hands over a finished PRD. Create a page in the **PRDs DB** (not the Issues DB): `mcp__notion__notion-create-pages`, parent = PRDs DB ID. Body = the PRD. Set the **Status** property to the `ready-for-agent` role string. (A PRD goes to the PRDs DB even though `/to-prd` says "publish to the project issue tracker".)

### publishIssues(artifacts[])

`/to-issues` hands over an approved set of vertical-slice issues, each with its category and state. Publish each as a page in the **Issues DB**: `mcp__notion__notion-create-pages`, parent = Issues DB ID. Set the title (required), the **Status** property to the mapped state-role string (default `ready-for-agent` for AFK slices unless instructed otherwise), and **Category** (`bug` / `enhancement`) when known. Body = the issue artifact.

Publish in dependency order (blockers first) so real page IDs exist before you reference them. For each issue's "Blocked by", set the **Blocked by** relation to the blocking issues' page IDs.

### publishHandoff(doc)

`/handoff` hands over a handoff document for a fresh session to pick up. Create a page in the **Handoffs DB**: `mcp__notion__notion-create-pages`, parent = Handoffs DB ID. Body = the handoff document. Return the page URL so the next session can reference it.

### fetchTicket(id | title)

`mcp__notion__notion-fetch` on the page ID. Child pages (e.g. an attached agent brief) come back in the response, so no separate lookup is needed. If given a title rather than an ID, resolve it via `listIssues` first.

### listIssues(filter)

`mcp__notion__notion-query-data-sources` against the Issues DB data source, filtering on the **Status** property. Present oldest-first per the triage buckets the caller asked for.

### comment(id, body)

`mcp__notion__notion-create-comment` on the page ID; body = the note in markdown, with the AI disclaimer prepended (see above).

### transitionState(id, role)

`mcp__notion__notion-update-page` on the page ID, setting the **Status** property to the mapped role string. The canonical-role → string mapping lives in `workflow-config.md`. The five canonical **state** roles: `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, `wontfix`. "Close as wontfix" = `transitionState(id, wontfix)` plus a `comment` explaining why.

### attachAgentBrief(id, brief)

When an issue moves to `ready-for-agent` / `ready-for-human`, create the brief as a **child page** of the issue (`mcp__notion__notion-create-pages` with `parent: { page_id: <issue page id> }`) so it has a stable URL and isn't truncated by comment-length limits. Body = the brief the content skill produced.

### applyTriageOutcome(id, outcome)

`/triage` hands over a resolved outcome — a target state, optional category, and the comment / brief body. Apply it as the combination of the operations above: `transitionState`, then `comment` and/or `attachAgentBrief` as the outcome requires, and close on `wontfix`.

## Adding a Select option (Epic and similar)

To append an option to a Select property without overwriting the existing ones:

1. `mcp__notion__notion-fetch` the DB ID and read `properties.<PropertyName>.select.options`.
2. `mcp__notion__notion-update-data-source` with `ALTER COLUMN "<PropertyName>" SET SELECT(...)` listing **all** existing options **plus** the new one.

Never use `ADD COLUMN` — that adds a new property, not a new option.

## Pull requests as a triage surface

**N/A — Notion has no pull requests.** `/triage` processes issues only for this repo. (The PR-as-request-surface flag applies only to GitHub / GitLab backends.)

## Error handling

If a Notion MCP call fails (auth, missing DB, schema mismatch), stop the operation and tell the user. Do not silently retry or fall back to another backend.
